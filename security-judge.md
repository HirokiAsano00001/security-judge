# Claude Code セキュリティ評価エージェント「security-judge」仕様書

## 1. システム概要
本エージェント「security-judge」は、Claude CodeのMCP（Model Context Protocol）ツールとして動作し、制作したアプリに対する自律型サイバー攻撃（Agentic Hacking）を実行する。
フロントエンドのUI制限を無視したAPIへの直接攻撃、ソースコードの静的解析、LLMへのプロンプトインジェクションを組み合わせなど、
最も厳格な基準でセキュリティ耐性をスコアリングする。単なるツール呼び出しではなく、完了条件を満たすまで自律的に攻撃と検証を繰り返すシステムとして機能する。


## 2. 評価実行フロー

### フェーズ1: アプリケーション・ペルソナの定義（事前ヒアリング）
（`/security-judge_goal` コマンドにより起動）
エージェント起動時、ターゲット層をヒアリングし、スコアリングの厳格度（減点係数）を動的に**0~10点**で評価する。
* **自分用（個人運用）:** 致命的な情報漏洩（環境変数やクラウド認証情報のハードコード等）のみ厳格に減点。
* **チーム用（特定の小規模グループ）:** BOLA/IDOR（他人のデータへのアクセス）を中程度の減点対象とする。
* **社内用（企業内クローズド）:** 権限昇格や内部ネットワーク情報の漏洩（SSRF等）を厳格に減点。
* **商用（一般公開・マネタイズ視野）:** 最高難易度（Zero Trust前提）。全てのバリデーションバイパス、XSS、LLMの脱獄に対して最大係数で減点・即時フェイル（0点）を適用する。

### フェーズ2: 情報収集（Reconnaissance）
* ソースコード全体のスキャン（環境変数、ハードコードされたシークレットの抽出）。
* バックエンド（Spring Boot等のコントローラー）定義からの全非公開エンドポイントの特定。
* フロントエンド（React等）に実装されたクライアント側のバリデーションロジックの特定。

### フェーズ3: 自律的攻撃実行（Exploitation）
* ブラウザのUI操作（DevTools等）を一切経由せず、HTTPクライアントツールを用いてバックエンドAPIへ直接エクスプロイトを送信する。
* エラー応答を受信した場合は、その内容から内部構造を推論し、最大3回までペイロードを自動修正して再攻撃を行う。

### フェーズ4: 採点とレポーティング
* ペルソナ基準に基づいた減点方式でスコアを算出し、**0~10点**で評価
再現可能な攻撃リクエスト（curlコマンド等）とともにレポートを出力する。

## 3. 実装するMCP Skill（ツール定義）

| ツール名 | 機能説明 | 厳格評価における役割 |
| :--- | :--- | :--- |
| `ask_target_persona` | ユーザーにアプリの用途（個人/チーム/社内/商用）を質問し、環境変数にセットする。 | 評価基準のベースラインを決定する。 |
| `analyze_sast_deep` | AST（抽象構文木）レベルでコードを解析し、危険な関数呼び出しやシークレットのハードコードを検知する。 | 開発者の実装の穴（キーのベタ書き等）を許容せず機械的に検知する。 |
| `fuzz_api_direct` | 指定されたAPIエンドポイントに対して、UI制限を無視した異常値・長文・改ざんパラメータをPOSTする。 | クライアント側のみのバリデーションを完全に無効化し、サーバー側の堅牢性を暴く。 |
| `test_bola_idor` | 取得した認証トークンを保持したまま、他ユーザーのIDやリソースURLに対してアクセスを試みる。 | 認可チェックの欠落（横方向の権限昇格）を検知する。 |
| `inject_llm_jailbreak` | AIチャット等の入力欄に対し、システムプロンプトの抽出や制限解除を狙うマルチターン（対話型）のプロンプトを送信する。 | LLM連携機能における境界防御の不備を検知する。 |

## 4. 最厳格スコアリング・ルーブリック（減点基準）
商用ペルソナを選択した場合、以下の事象が1つでも確認された機能はセキュリティスコアを「0点」とし、総合評価から大幅に減点する。

* **A. バックエンド検証の欠落（DevToolsバイパス）**
  * 検知条件: React等のフロントエンドで入力制限（文字数、型、非活性化）を行っているが、`fuzz_api_direct` によるAPI直接攻撃で異常値がそのままデータベースに保存、または処理された場合。
  * 評価: 即時フェイル。UI上の制限はセキュリティ対策とみなさない。
* **B. 認可制御の欠落（BOLA/IDOR）**
  * 検知条件: ログイン機構は存在するが、APIパラメータや隠しパラメータ（`isAdmin=true` 等）を改ざんしたリクエストが成功した場合。
  * 評価: 即時フェイル。認証（Authentication）と認可（Authorization）の混同とみなす。
* **C. 情報漏洩（Verbose Errors / Hardcoded Secrets）**
  * 検知条件: 意図的なエラーを引き起こした際、スタックトレース、SQL構文、フレームワーク（Tomcat等）のバージョンがHTTPレスポンスに含まれた場合。またはコード内にAPIキーが存在する場合。
  * 評価: 即時フェイル。攻撃者への内部構造の開示とみなす。
* **D. LLMガードレール突破**
  * 検知条件: `inject_llm_jailbreak` により、AIがシステムプロンプトの一部を漏洩する、または開発者が意図しないタスク（外部URLの読み込み等）を実行した場合。
  * 評価: 段階的減点（ペルソナにより変動）。商用・社内用の場合は、AIに付与された権限の範囲に応じて致命的脆弱性として扱う。

## 5. 自律実行アーキテクチャ
本エージェントは、Claude Codeの Dynamic Workflow を用いて自律的にオーケストレーションされ、人間の介入なしに攻撃ミッションを完結させる。

* **タスクの自動オーケストレーション:** メインプロセスは、フェーズ2（情報収集）からフェーズ3（エクスプロイト実行）、フェーズ4（採点・レポート生成）までを1つのワークフローとして管理する。
* **並列実行:** 静的解析、APIへのFuzzing、LLM脱獄試行などの各Skillは、サブエージェントとしてバックグラウンドで並行実行され、効率的かつ網羅的にアプリケーションの堅牢性を検証する。
* **自己修復・継続攻撃:** エラー応答を受けた際のペイロード自動修正（最大3回）や、別エンドポイントへの攻撃遷移は、人間への進捗確認を挟まずに自律判断で実行される。

## 6. カスタムコマンド定義
本エージェントの操作、自律実行のトリガー、および開発者支援として以下の専用コマンドを定義する。

### コマンド: `/security-judge_goal`
* **機能説明:** セキュリティ評価の自律実行を開始するためのメインコマンド。
* **実行フロー:**
  1. 本コマンドを実行すると、まずフェーズ1（アプリケーション・ペルソナの定義）がトリガーされ、ユーザーに対してターゲット層（自分用 / チーム用 / 社内用 / 商用）の事前ヒアリングを行う。
  2. ユーザーの回答を受け取り、評価基準（減点係数）を内部コンテキストにセットする。
  3. 設定完了後、人間の介入を必要とせず、フェーズ2（情報収集）からフェーズ4（採点・レポーティング）までの自律ハッキングループを自動的に開始・完結させる。

### コマンド: `/security-judge_remedy`
* **機能説明:** ハッキング完了後、または実行プロセス中に、検出された脆弱性に対する防御策・修正案を提示するコマンド。
* **実行フロー:**
  1. `security-judge` が検出した脆弱性（BOLA、API入力検証欠落、シークレット漏洩など）の情報を読み込む。
  2. 該当箇所の特定（ファイル名・行数・エンドポイント）を行う。
  3. 具体的な修正コード案（Before / After）や、フレームワーク（Spring Boot等）に応じたセキュア設定の変更例を出力する。コードの直接書き換えは行わず、修正案の提示に留める。
